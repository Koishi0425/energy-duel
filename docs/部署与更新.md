# 《娇斯拉大战贡刚》部署与更新

本项目的生产形态是一个 Node.js 进程：Colyseus 同时承载 HTTP API、WebSocket
房间和 `client/dist` 静态资源。Docker Compose 默认只启动一个 `game` 服务，账号
文件保存在独立的 `energy-duel-data` 命名卷中。

生产镜像基于 Node.js 22。Dockerfile 会先完成整个项目构建，再串行安装运行时
依赖，避免小内存服务器同时执行两次 `npm ci`。

## 1. 部署前准备

服务器建议使用 Linux，并安装 Docker Engine、Docker Compose 插件和 Git。最低
需要开放一个供反向代理访问的本地端口；面向公网部署时建议准备域名并启用 HTTPS。

```bash
git clone <你的仓库地址> energy-duel
cd energy-duel
cp .env.example .env
```

`.env` 可配置：

```dotenv
ENERGY_DUEL_PORT=2567
ENERGY_DUEL_TAG=local
```

首次构建并启动：

```bash
docker compose build --pull
docker compose up -d
docker compose ps
curl --fail http://127.0.0.1:2567/api/health
```

查看实时日志：

```bash
docker compose logs -f --tail=200 game
```

也可以使用 `npm run docker:build`、`npm run docker:up`、
`npm run docker:logs` 和 `npm run docker:down` 执行对应操作。

> `docker compose down` 不会删除账号卷；不要执行 `docker compose down -v`，
> 除非确实要删除全部本地账号映射。

## 2. 域名、HTTPS 与 WebSocket

测试环境可以直接访问 `http://服务器IP:2567`。生产环境建议让 Nginx、Caddy 或
云负载均衡器终止 TLS，再将同一个域名的 HTTP 和 WebSocket 流量代理到
`127.0.0.1:2567`。这样客户端使用同源地址，无需另外配置 WebSocket 域名。

仓库提供了 `deploy/nginx-energy-duel.conf.example`。复制后修改
`game.example.com`，再使用服务器现有的证书工具启用 443/HTTPS。代理必须保留
`Upgrade` 和 `Connection` 请求头。建议只让防火墙向本机开放 2567，对公网仅开放
80/443。

部署完成后检查：

- `https://你的域名/api/health` 返回 `status: ok`；
- 登录、创建房间、快速加入以及准备状态能正常同步；
- 两个不同网络的浏览器可以留在同一房间，浏览器控制台没有 WebSocket 错误；
- 容器重启后原有用户名仍可重新建立会话。

## 3. 日常更新

当前房间、临时会话令牌和断线重连席位都保存在进程内存中。替换容器会中断正在
进行的对局，因此应先通知玩家并选择维护窗口。

推荐每次使用 Git 提交号作为镜像标签，便于回滚：

```bash
git pull --ff-only
export ENERGY_DUEL_TAG="$(git rev-parse --short HEAD)"
docker compose build --pull game
docker compose up -d --no-deps game
docker compose ps
curl --fail http://127.0.0.1:2567/api/health
docker compose logs --tail=200 game
```

确认正常后，把 `.env` 中的 `ENERGY_DUEL_TAG` 改为本次提交号，使服务器重启后仍
使用同一标签。Compose 只会替换 `game` 容器，不会删除命名卷里的账号数据。

若新版本异常，可将 `.env` 的标签改回之前的提交号，然后运行：

```bash
docker compose up -d --no-deps game
```

回滚要求旧镜像仍保留在服务器上。发布前不要立即执行 `docker image prune`。
数据库或配置格式以后发生不可逆迁移时，还需要随版本提供专门的迁移与回滚脚本。

## 4. 账号数据备份与恢复

更新前可备份整个命名卷：

```bash
mkdir -p backups
docker run --rm \
  -v energy-duel-data:/data:ro \
  -v "$PWD/backups:/backup" \
  alpine tar czf "/backup/energy-duel-data-$(date +%Y%m%d-%H%M%S).tar.gz" -C /data .
```

恢复时先停止服务，并确认备份文件名：

```bash
docker compose stop game
docker run --rm \
  -v energy-duel-data:/data \
  -v "$PWD/backups:/backup:ro" \
  alpine sh -c 'rm -rf /data/* && tar xzf /backup/<备份文件名>.tar.gz -C /data'
docker compose start game
```

恢复操作会覆盖现有账号数据，应先额外保留当前卷的备份。账号只有用户名映射；会话
令牌不会持久化，重启后客户端会用保存的用户名自动申请新会话。

## 5. 当前扩容边界

请保持单个 `game` 实例，不要直接执行 `docker compose up --scale game=2`。当前会话
令牌、房间和 Colyseus Presence 都是进程内状态，多实例会导致同一个房间号落到不同
进程，表现为“有人加入不了、却又能创建同名房间”。

未来要水平扩容，至少需要：

- Redis Presence 和适合多进程的 Colyseus Driver；
- 将会话令牌和账号数据迁移到共享数据库或 Redis；
- 在网关统一进行房间发现与路由，并正确转发 WebSocket；
- 明确对局迁移、优雅停机和版本兼容策略。

在完成这些改造前，纵向升级单台服务器比增加副本更安全。
