# 资源与素材管理

## 三类资源必须分开

1. **美术导入源文件**：放在 `art-source/runtime-imports/`，保留收到的无损 PNG，但不进入 Docker 构建上下文。
2. **项目运行时素材**：由 `npm run assets:optimize` 生成到 `client/public/assets/`，会随客户端部署。
3. **游戏配置和资源 ID**：放在 `shared/config/`。代码和存档只引用稳定 ID，不依赖美术文件名。
4. **用户上传数据**：放在 `server/data/`。头像位于 `server/data/avatars/`，不得提交到 Git。

PSD、CLIP、AE、Blender 等源工程不要放进 `client/public/`。建议单独使用美术源文件仓库、共享网盘或 Git LFS；导出后再把压缩后的成品加入本项目。

## 推荐目录

```text
art-source/runtime-imports/
├─ characters/<character-id>/portrait.png
└─ profiles/{nameplates,titles}/<source-file>.png

client/public/assets/
├─ characters/
│  └─ <character-id>/
│     └─ <form-id>/
│        ├─ portrait.<content-hash>.webp
│        └─ preview.<content-hash>.webp
├─ profiles/
│  ├─ defaults/avatar.webp
│  ├─ nameplates/<nameplate-id>/frame.webp
│  ├─ nameplates/<nameplate-id>/thumbnail.webp
│  ├─ titles/<title-id>/badge.webp
│  ├─ ranks/<rank-id>/emblem.webp
│  └─ achievements/<achievement-id>/icon.webp
├─ ui/<feature-id>/...
├─ vfx/<effect-id>/...
├─ manifests/assets.json
└─ audio/
   ├─ bgm/
   ├─ sfx/
   └─ voice/

server/data/
├─ users.json
└─ avatars/<account-id>.webp
```

角色完整立绘最长边为 1024px，棋盘预览最长边为 384px，均保留透明通道。泥岩与梦魇等透明底 PNG 会在转换后继续保留 Alpha。客户端首屏只并发加载最多 3 张预览图；完整立绘留给详情等确实需要高分辨率的界面。

## 命名规则

- 文件夹和文件只使用小写 ASCII、数字和连字符，例如 `sovereign-blade`。
- 配置 ID 一经上线不要改名。显示名称可以修改，ID 用于存档、解锁和历史记录。
- 不在文件名中加入 `final`、`new`、日期或版本号。浏览器缓存通过构建哈希或配置版本处理。
- 同一类素材保持统一尺寸和安全边距。

## 推荐格式和尺寸

| 类型 | 建议规格 | 格式 |
|---|---|---|
| 角色立绘 | 长边不超过 2048px，透明背景 | WebP；需要无损细节时 PNG |
| 玩家头像 | 服务端统一裁剪为 512×512 | WebP |
| 姓名框 | 当前统一 720×116，透明背景 | WebP/PNG |
| 姓名框缩略图 | 最长边 320px | WebP |
| 称号徽章 | 建议 256×64 或矢量 | WebP/SVG |
| 段位徽章 | 512×512，主体居中 | WebP |
| 成就图标 | 256×256 | WebP |
| UI 图标 | 优先矢量 | SVG |
| 音效 | 短音效 | OGG，必要时提供 MP3 回退 |
| 背景音乐 | 循环点记录在清单中 | OGG |

不要把未经压缩的 4K PNG、源工程或重复导出版本直接放进运行时目录。

## 姓名框的安全区域

姓名框的 `frame.webp` 是整张玩家横幅背景图片，不是昵称文字周围的局部边框。姓名框会适配桌面和移动端，素材应按以下区域设计：

- 左侧头像装饰区；
- 头像右侧的三行文字安全区，依次用于 Rating、昵称（含段位徽标）和称号；
- 三行信息右侧上方的等级徽章安全区；
- 最右侧可展示主题图案的装饰区。

关键装饰不要压住头像、三行文字和等级徽章，移动端裁切时还应保证左侧信息完整。姓名框始终按 720:116 比例显示。代码只保存 `nameplateId`；素材清单负责把稳定 ID 映射到带内容哈希的文件，替换素材不会修改玩家资料。

## 转换、清单与缓存

- 运行 `npm run assets:optimize` 会先清理管线拥有的旧生成目录，再生成 WebP 全图、预览图和 `assets.json` 清单，避免废弃哈希文件积累。
- 文件内容变化才会产生新的 12 位内容哈希。带哈希的立绘、姓名框、称号和 Vite JS/CSS 使用一年不可变缓存；入口 HTML 与清单使用协商缓存，确保发布后能发现新文件。
- 尚未确定游戏内 ID 和解锁方式的姓名框、称号会以 `unregistered` 状态进入清单，不会自动开放给玩家。确定映射后再登记到 `PROFILE_NAMEPLATES` 或 `PROFILE_TITLES`。
- `art-source/` 会提交到源码仓库以保留可复现输入，但不会复制进生产镜像。未来源工程明显变大时，应迁移到 Git LFS 或独立美术库。

## 素材登记流程

1. 先确定稳定 ID，例如 `nameplate-season-one`。
2. 将源工程存入外部美术库，并记录作者、授权和源文件位置。
3. 将无损导出文件放入 `art-source/runtime-imports/`，运行 `npm run assets:optimize`。
4. 在对应配置中登记清单里的稳定 ID、URL、显示名称、稀有度和解锁条件。
5. 运行配置测试、客户端构建，并检查桌面和手机布局。
6. 替换已上线文件时更新配置引用；内容哈希会让浏览器获取新资源。

建议维护一份素材清单，至少包含：`id`、类型、运行时路径、源文件位置、作者、授权、尺寸、状态、解锁来源和备注。

## 用户头像

用户原图只在浏览器中用于选择裁剪范围。客户端输出 512×512 WebP，服务端验证格式和 1MB 大小限制后保存。不要把用户头像写入 `game.json`、同步到 Colyseus 房间状态或提交到 Git；房间状态只应携带账号或头像版本标识。
