# 严格角色设计工作流

处理 `$ARGUMENTS` 指定的角色、初稿或文件；未传参数时处理当前对话中的设计稿。

## 阶段 0：确定任务边界

将请求归入以下一种或多种范围：

1. 审核初稿；
2. 生成新角色文档；
3. 修订现有角色文档；
4. 对照当前实现；
5. 实现配置和代码。

默认执行 1–4。只有用户明确要求实现时才执行 5。

## 阶段 1：加载依据

必须读取：

- 根目录 `AGENTS.md` 和 `CLAUDE.md`；
- `docs/基础规则手册.md`；
- `docs/角色信息手册.md`；
- 目标角色文档（若存在）。

对照已实现行为时，再读取：

- `shared/config/game.json` 中相关角色、行动、Buff、被动和资源；
- `server/src/game/RoundResolver.ts` 中相关处理器；
- 相关测试。

不要把旧角色文档未经核对的内容当作用户当前意图。

## 阶段 2：建立规则清单

逐个技能提取：

- 名称、分类、消耗、等级、速度、伤害类型；
- 使用条件、合法对象、数量、选择时机、选择次数；
- 执行顺序、结算时点、自动作用范围；
- Buff 归属、层数、上限、持续、刷新、消耗、移除和重置；
- 每回合、每次行动、每名玩家等触发频率；
- 无合法对象、对象死亡或移动、资源不足、同时结算时的处理；
- 变身、死亡、位移、免疫、后发和角色切换交互。

缺少且无法唯一推出的项目记为规则缺口。

## 阶段 3：通过五项审核门

### A. 完整性

- 表格字段齐全。
- 每个状态有获得、作用和结束方式。
- 每个成长机制有上限或明确无上限。
- 每个次数机制说明获得与消耗。

### B. 唯一解释

- 同一句话不能产生两种不同结算。
- 区分“选择”与“自动影响”。
- 区分玩家、实体、目标、地块和角色。
- 区分依次执行、同时执行和条件执行。
- 持续时间说明开始、结束以及是否包含当前回合。

### C. 系统一致性

- 符合基础等级差、生命状态、速度和后发规则。
- 明确 Buff 是角色级还是玩家级。
- 明确切换角色后状态是否保留和计时。
- 不与已有同名术语产生新含义。

### D. 设计健康度

- 检查无限资源、无限行动和不可终止循环。
- 检查严格优于或严格劣于既有技能的情况。
- 检查多人局中的无合法对象、重复命中和范围膨胀。
- 检查是否存在合理反制窗口。

只提出平衡建议，不擅自改数值。

### E. 可实现性

- 标出当前配置模型无法表达的效果。
- 标出需要新实体类型、新选择协议、新结算阶段或新同步字段的机制。
- 不因当前代码难以实现而偷偷改写设计。

## 阶段 4：歧义闸门

将问题分为：

- **阻断项**：不同答案会改变合法目标、资源支付、伤害、死亡、行动顺序、持续时间或状态归属。
- **非阻断项**：命名、风味文字、章节顺序等不改变结算。

存在阻断项时：

1. 列出问题；
2. 给出推荐解释；
3. 说明各解释的结算差异；
4. 在用户确认前，不把推荐解释写成最终规则。

若用户明确要求直接完成，采用新增机制最少的保守解释，并在审核摘要中列出假设。非阻断项可按仓库现有风格处理。

## 阶段 5：改写为受控文法

遵守以下规则：

- 每句只表达一个主要规则动作。
- 按“触发或选择 → 执行动作 → 持续或结束”排序。
- 使用阿拉伯数字：`1 名`、`0.5 级`、`至少 2 层`。
- `可以` 表示可选；`必须` 表示强制。
- `然后` 表示顺序；`同时` 表示同一结算点；`若……则……` 表示条件。
- 使用有唯一先行对象的“该玩家”“该地块”“此次攻击”，不用指向不明的“其”“对方”“它们”。
- 不使用“附近”“单体”“敌人”“全场目标”“一段时间”“有概率”等不完整规则词。

### 固定目标句式

- `1 名其他存活玩家（预选）`
- `1 名其他存活玩家（后发）`
- `其他存活玩家，选择 2 次（预选，可重复）`
- `自身格及顺、逆时针各 2 格内的所有其他存活玩家（自动）`
- `本回合使用“吸”的所有玩家（自动）`
- `1 个顺或逆时针相邻空地块（预选）`
- `自身（自动）`

若作用对象是非玩家实体，将“玩家”替换为“实体”；两者都可以选择时使用“目标”。不要在角色文档中额外插入目标类型教程。

### 固定状态句式

- `获得 1 层“X”，至多为 4 层。`
- `消耗 1 层“X”；然后……。`
- `若已拥有“X”，将其持续时间重置为 3 回合。`
- `使用后，将“X”层数重置为 0。`
- `持有至少 1 层“X”时可以使用。`
- `每回合至多触发 1 次。`

只有设计明确包含相应规则时才使用模板，不得由模板反向创造规则。

## 阶段 6：生成角色文档

文档路径为 `docs/角色/<角色名>.md`。沿用仓库现有结构：

1. 标题与一句话定位；
2. 基础规则和附录链接；
3. 属性；
4. 被动技能（如有）；
5. 变身收益；
6. 核心机制（仅共享机制需要）；
7. 专属技能；
8. 设计定位。

每个专属技能使用：

```markdown
### 技能名

| 属性 | 值 |
|------|-----|
| 分类 | 攻击 / 防御 / 资源 / 特殊 |
| 消耗 | 明确资源 |
| 速度 | 数值 |
| 等级 | 数值或严格计算式 |
| 伤害类型 | 仅攻击需要 |
| 目标 | 固定目标句式 |
| 解锁条件 | 仅条件技能需要 |

**结算说明**：

1. 按执行顺序描述效果。
2. 每项只描述一个主要动作。
```

删除不适用的章节和表格行，不写无意义占位符。设计定位只说明玩法循环、优势、风险和反制窗口，不在其中新增规则。

## 阶段 7：交付与同步检查

默认交付：

1. **审核摘要**：规则缺口、冲突、实现风险、平衡风险和采用的假设；
2. **规范化文档**：完整 Markdown 或已写入的文件路径；
3. **一致性差异**：文档、教程、配置和服务端之间的差异；
4. **待实现项**：仅列出，不自动实现。

审核摘要不得写进角色文档。

修改现有角色后检查：

- `docs/角色信息手册.md` 的索引与定位；
- `docs/基础规则手册.md` 的公共术语；
- `client/src/content/gameGuide.ts` 的短篇介绍；
- `shared/config/game.json` 和服务端行为。

## 完成检查

交付前逐项确认：

- [ ] 所有阻断歧义已确认或明确标注假设；
- [ ] 每个技能都能唯一确定合法对象和执行时机；
- [ ] 所有层数、次数、持续时间和重置条件完整；
- [ ] 规则正文未使用模糊代词或不完整范围词；
- [ ] 审核意见没有混入最终角色文档；
- [ ] 未经授权没有修改配置或代码；
- [ ] 已列出与当前实现的差异。
